# Task Advisor — AI-подбор задач

Документ описывает сервис `App\AI\TaskAdvisor`, который подбирает оптимальный набор задач из списка пользователя под заданное свободное время и контекст. Это сердце фичи `/free` — главного пользовательского сценария проекта.

## Роль и отличие от TaskParser

| | TaskParser | TaskAdvisor |
|---|---|---|
| Задача | Извлечение структуры из текста | Рассуждение о подборе |
| Вход | Одно сообщение пользователя | Список задач + контекст |
| Выход | ParsedTaskDTO | TaskSuggestionDTO |
| Модель | haiku (быстрая экстракция) | sonnet (рассуждение) |
| Temperature | 0.0 (детерминизм) | 0.3 (вариативность для reroll) |

Парсинг — это структурное извлечение: текст → поля. Подбор — это рассуждение с учётом многих факторов: приоритет, дедлайн, контекст места, группировка по маршруту, баланс времени. Haiku для парсинга достаточно умна, но для подбора нужна Sonnet.

## Почему Sonnet, а не Haiku

Подбор требует одновременного рассуждения о:
- Приоритетах и срочности (задачи с близким дедлайном и urgent — первыми)
- Контексте места (дом vs улица vs офис vs дача)
- Группировке по маршруту (ищет совпадения в description: адреса, «рядом», «по пути»)
- Балансе времени (сумма ≤ available, с запасом 10-15%)
- Порядке выполнения (последовательность, а не просто множество)

Haiku путалась на задачах с несколькими конкурирующими критериями. Sonnet держит все факторы в голове и выдаёт осмысленный reasoning. Стоимость оправдана — `/free` вызывается редко (несколько раз в день), в отличие от парсинга.

## System prompt

```
Ты AI-помощник по управлению задачами. Пользователь сообщил, сколько у него
свободного времени и где он находится. Твоя задача — подобрать из его открытых
задач оптимальный набор, который он реально сможет сделать прямо сейчас.

Текущее время: {nowLocal в формате Y-m-d H:i (l)}
Часовой пояс: {IANA tz name}
Доступное время: {availableMinutes} минут
{контекстная строка}
{excluded_ids — при reroll}

Отвечай строго JSON...
{JSON schema}

Правила подбора:
1. Приоритет и срочность
2. Уложиться во время
3. Контекст места
4. Группировка по месту/маршруту
5. Зависимости
6. Порядок выполнения
7. Совет (tip)
8. Пустой результат
9. Переполнение
```

### Плейсхолдеры

- `{nowLocal}` — текущее время в зоне пользователя (`$now->setTimezone($userTz)`)
- `{tz}` — IANA-имя из `$user->getTimezone()`
- `{availableMinutes}` — int из `/free <время>`
- `{контекстная строка}` — либо «Контекст пользователя: {context}», либо «Контекст не указан»
- `{excluded_ids}` — при reroll: список task_ids, которые уже отвергнуты

## User prompt

JSON-массив задач в `<available_tasks>...</available_tasks>`. Поля каждой задачи:

- `id` — UUID
- `title`, `description`
- `deadline_iso` (ISO 8601 в зоне пользователя, либо null)
- `estimated_minutes`
- `priority` (low/medium/high/urgent)
- `context_codes` (массив)
- `created_at_iso`
- `is_overdue` (bool, посчитано в PHP — удобнее для AI чем вычислять)

## Формат ответа

```json
{
  "user_summary": "Короткое дружеское объяснение (≤300 символов, без перечисления отвергнутых)",
  "internal_reasoning": "Подробный анализ для логов: что выбрано/отвергнуто и почему",
  "suggestions": [
    {
      "task_id": "019d9774-...",
      "order": 1,
      "tip": "Совет или null",
      "estimated_minutes": 20
    }
  ],
  "total_estimated_minutes": 35,
  "no_match_reason": null
}
```

Если подходящих нет — `suggestions: []` + `no_match_reason` с объяснением.

### Разделение user_summary и internal_reasoning

Изначально был один `reasoning`, но он получался либо слишком коротким (мало пользы для отладки), либо слишком длинным (пользователь видел перечисление всех отвергнутых задач с причинами — шум).

Теперь два поля:

- **`user_summary`** идёт в Telegram-ответ в блоке `💭`. Требования: 1-3 предложения, разговорный тон, без перечисления отвергнутых задач, без нумерации пунктов, ≤ 300 символов. Если AI выдал длиннее — TaskAdvisor обрезает до 299 символов с `…` и логирует warning `AI did not comply with summary length limit`.
  - ✅ «Взял срочные outdoor-задачи — корм и штаны можно совместить в один выход.»
  - ❌ «Выбрал 2 задачи. Отверг дачные (пользователь дома), видеокурс (не влезает в 2 часа)...»

- **`internal_reasoning`** НЕ идёт в UI. Логируется через Monolog в `FreeHandler::suggestWithRetry()`:
  ```php
  $this->logger->info('TaskAdvisor reasoning', [
      'user_id' => ..., 'available_minutes' => ..., 'context' => ...,
      'suggestions_count' => ..., 'internal_reasoning' => ...,
  ]);
  ```
  Модель может писать подробно: каждую задачу взвесил, отвергнутые с причинами, дилеммы. Это не видит пользователь — только для отладки и оценки качества подбора.

В DTO: `TaskSuggestionDTO->userSummary` (в Telegram) и `->internalReasoning` (в логах).

## Правила подбора

1. **Приоритет и срочность.** urgent или is_overdue — почти всегда в предложении (если контекст подходит). high+дедлайн в ближайшие 24-48ч — в первую очередь.
2. **Уложиться во время.** Сумма estimated_minutes ≤ availableMinutes. Без оценки — прикинь по смыслу. Лучше -10-15%, чем перегруз.
3. **Контекст места.** «Дома» → никаких outdoor/at_dacha/at_office. «На улице» → никаких focused/needs_internet обычно. Контекст не указан → любые.
4. **Группировка по маршруту.** Ищет в description совпадения: адреса, «рядом», «по пути», «заодно». Если можно совместить — предлагает вместе и объясняет в reasoning/tip.
5. **Зависимости.** Заблокированные не приходят (handler фильтрует). Если попадут — игнорировать.
6. **Порядок выполнения.** Оптимальная последовательность: по пути → дом, короткие → сложные. Как человек, планирующий день.
7. **Совет (tip).** Короткий практический совет к каждой задаче. Если самоочевидно — null. Не повторять title.
8. **Пустой результат.** Если ничего не подходит — suggestions: [] + объяснение в no_match_reason.
9. **Переполнение.** Если задач больше чем влезает — выбрать самые важные, упомянуть сколько ещё подходит.

## Валидация ответа

В `TaskAdvisor::parseResponse()`:

- `task_id` каждого suggestion должен быть среди переданных → иначе игнорируется с warning
- Дубли order сортируются просто по возрастанию (не проверяем уникальность жёстко — AI иногда ошибается с нумерацией, но порядок в массиве корректный)
- `total_estimated_minutes > availableMinutes` → обрезаем предложение до первого, которое не влезает, логируем warning
- Тройной JSON fallback (как в TaskParser): прямой decode → ````json ... ` `` ` → первый `{ ... }` в тексте

## Inline-кнопки и хранение состояния

`/free` возвращает три кнопки: **✅ Беру!**, **🔄 Другие варианты**, **❌ Не сейчас**.

**Проблема**: Telegram ограничивает callback_data 64 байтами. Несколько UUID туда не влезают (36 байт × N + мета ≥ 72 для двух задач).

**Решение**: состояние хранится в Redis, в callback_data — короткий ключ.

```
free:<12-hex>:take    // ~22 байта
free:<12-hex>:reroll  // ~24 байта
free:<12-hex>:dismiss // ~25 байт
```

Redis-key: `free:<12-hex>`. Ключ генерируется `bin2hex(random_bytes(6))`. TTL **1 час** — достаточно чтобы пользователь посмотрел и выбрал, не накапливает мусор.

Payload:
```json
{
  "user_id": "<uuid>",
  "task_ids": ["<uuid>", ...],
  "minutes": 120,
  "context": "дома",
  "excluded_ids": ["<uuid>", ...],
  "reroll_count": 0
}
```

Сервис-обёртка: `App\Service\FreeSuggestionStore` — инкапсулирует Redis-доступ через `ext-redis`. Конструктор принимает `REDIS_URL`, парсит host/port.

## Лимит rerolls

Максимум **3 reroll'а подряд**. При 4-м — сообщение «Больше ничего не подобрать».

Механика: `reroll_count` в Redis инкрементируется при каждом `reroll`. `excluded_ids` накапливаются — при reroll в них добавляются все `task_ids` из текущего предложения. Следующий вызов advisor получает полный список исключений.

## Действия кнопок

| Кнопка | Действие |
|---|---|
| ✅ Беру! | Все task_ids → status `IN_PROGRESS`, удалить ключ Redis, edit message «🚀 Удачи! Когда закончишь — /done» |
| 🔄 Другие варианты | Вызвать `TaskAdvisor::suggest()` с `excludeTaskIds = previous + accumulated`, reroll_count++, сохранить новый ключ, edit message |
| ❌ Не сейчас | Удалить ключ, edit message «Ок, отдыхай! 🌴» |

Все callback'и начинаются с `$bot->answerCallbackQuery()` чтобы убрать «загрузку» на кнопке.

## Retry политика

В `FreeHandler::suggestWithRetry()` — та же схема что у TaskParser:

| Исключение | Максимум retry | Backoff |
|---|---|---|
| `ClaudeTransientException` | 2 | 1s, 3s |
| `ClaudeRateLimitException` | 1 | `retryAfter` или 5s |
| `ClaudeClientException` | 0 | сразу fallback |

При полном отказе — `editMessageText` с «⚠️ Не получилось подобрать задачу, попробуй ещё раз через минуту».

## Известные ограничения (отложено)

- **Нет доступа к календарю пользователя.** AI не знает про запланированные встречи, чужие дедлайны. Можно будет добавить через Google Calendar API.
- **Нет истории привычек.** AI не знает что пользователь «всегда откладывает задачи про бухгалтерию» или «лучше работает утром». Можно собирать эти паттерны в отдельную таблицу и добавлять в system prompt.
- **Нет геолокации.** «На улице» — это общий контекст. Точное местоположение и поиск «по пути» (с картами) — отдельная большая тема.
- **Нет real-time погоды.** «На улице в дождь» vs «на улице в солнце» — разные сценарии. Отложено.
- **Группировка по маршруту — чисто текстовая.** AI ищет совпадения в description. Нет геокодирования адресов.
